# T-092: Toast-система (in-app уведомления)

**Phase:** 5 — Полировка
**Model:** Sonnet 4.6 · `medium`
**Estimate:** ~1.5-2 часа

## Контекст

В проекте есть native OS notifications через Tauri (`lib/host/notify.ts`) и инвазивные `window.confirm` / `window.alert` для подтверждений. Но **in-app toast'ов нет** — мелкие user-feedback сообщения сейчас идут в `console.error` или как inline-ошибки.

Конкретные места, где toast'ы нужны:
- `Message.tsx` — edit-window-expired ошибка от backend (сейчас просто `console.error` в `ChatScreen.tsx:160`).
- `useScreenShare.ts` — уже использует `toast(...)` вызов которого нет (см. `tasks/T-050.md` контекст).
- `ChatScreen.tsx:203` — `window.confirm('удалить сообщение?')` стоит заменить на toast с undo (как в Gmail).
- Будущие фичи: «снимок отправлен», «не удалось загрузить файл», «пароль изменён, сессии сброшены».

## Файлы в скоупе

- `packages/polly/src/lib/toast.ts` (новый — общий API)
- `packages/polly/src/components/ToastHost.tsx` (новый — рендер контейнера)
- `packages/polly/src/app/Shell.tsx` (вмонтировать ToastHost)
- `packages/polly/src/features/chat/ChatScreen.tsx` (заменить console.error и confirm)
- `packages/polly/src/features/voice/useScreenShare.ts` (использовать toast вместо комментариев-заглушек)
- `packages/polly/src/features/files/upload.ts` (toast при failure)

## Цель

Один универсальный `toast.info / success / error / undo` API, поверх легковесной зависимости или самописного решения. Все edge-error'ы дают visible feedback вместо тишины.

## Что сделать

1. **Выбор стратегии**: `sonner` (~5 KB minzip, имеет undo и promise-toast) или самописный hook на 60 строк. Рекомендую `sonner` — экономит время.
2. **`lib/toast.ts`**:
   ```ts
   export const toast = {
     info(msg, opts?),
     success(msg, opts?),
     error(msg, opts?),
     undo(msg, onUndo, opts?), // показывает кнопку «отменить»
     promise<T>(p: Promise<T>, { loading, success, error }), // обёртка для async
   }
   ```
3. **ToastHost.tsx**: контейнер `<Toaster>` от sonner с настройками — position `bottom-right`, theme берётся из `useThemeStore`, стилизация через CSS classes под KD-палитру (`--kd-panel`, `--kd-text`, `--kd-border`).
4. **Замены**:
   - `ChatScreen.tsx:160` (edit fail): `toast.error('не получилось изменить — ' + err.message)`.
   - `ChatScreen.tsx:160` если `err.code === 'edit-window-expired'`: `toast.error('прошло больше 30 дней — изменить нельзя')`.
   - `ChatScreen.tsx:203` (delete confirm): заменить `window.confirm` на optimistic delete + `toast.undo('сообщение удалено', () => restoreMessage(id))`. 5-секундное окно для undo.
   - `useScreenShare.ts`: `toast.error('не удалось захватить экран')` при `NotReadableError`.
   - `files/upload.ts`: toast при HTTP-ошибке загрузки.
5. **Доступность**:
   - `role='status'` для info/success, `role='alert'` для error.
   - `aria-live='polite'` для info, `'assertive'` для error.
   - sonner делает это сам — проверить.

## Acceptance criteria (DoD)

- [ ] Edit чужого сообщения старше 30 дней (через editing race) → toast «прошло больше 30 дней»
- [ ] Delete сообщения → исчезает сразу, toast с «отменить»; клик «отменить» — восстанавливает
- [ ] Toast'ы переключают тему вместе с приложением (`data-theme="dark"`)
- [ ] Один и тот же error не дублируется (sonner делает dedup по id)
- [ ] Screen-reader озвучивает toast при появлении (ручная проверка через NVDA/JAWS — опционально)
- [ ] `pnpm typecheck` чистый

## Out of scope

- Push-уведомления (native OS) — это `lib/host/notify.ts`, остаётся отдельным каналом.
- Persisted notifications history — у нас Inbox для этого.
- Rich content в toast (картинки, кнопки 3+) — простой text + одна action-кнопка достаточно.

## Подсказки

- `sonner` имеет typed `toast()` функцию, можно сразу заменить наш wrapper его API'ем — но wrapper полезен для миграции на самописный позже.
- Undo-pattern требует optimistic update в коде вызывающего. Не делай toast.undo для опасных операций (изменение пароля, удаление сервера) — там нужны явные confirm.
- В Tauri toast `position: bottom-right` может пересекаться с native tray notification. Проверь, не дублируется ли визуально на одном и том же event.
