# T-016: REST — история сообщений (cursor) и отправка

**Phase:** 1 — Текстовый дом
**Model:** Sonnet 4.6 · `medium`
**Estimate:** ~2 часа

## Контекст
Пагинация — cursor-based (по message.id), а не offset-based. Это нужно для бесконечного скролла вверх без сдвигов при новых сообщениях.

## Файлы в скоупе
- `packages/speedy/src/db/schema.ts` (таблица `messages`)
- `packages/speedy/src/routes/messages.ts` (новый)
- `packages/ginzu/src/api-types.ts`

## Что сделать
1. **Таблица `messages`**:
   - id uuid pk (использовать `uuidv7()` — упорядочены по времени!)
   - channelId fk, authorId fk
   - content text not null
   - replyToId uuid nullable, fk self
   - createdAt, editedAt (nullable), deletedAt (nullable — soft delete)
   - Индекс composite (channelId, id DESC) для пагинации
2. **`GET /api/channels/:channelId/messages`**:
   - query: `before?: uuid` (cursor), `limit?: int` (default 50, max 100)
   - SQL: `WHERE channel_id = $1 AND id < $cursor AND deleted_at IS NULL ORDER BY id DESC LIMIT $limit + 1`
   - Если вернулось `limit + 1` — есть ещё, `nextCursor = последняя строка.id`
   - Ответ: `{ messages: Message[], nextCursor: string | null }`. Сообщения в порядке от **старого к новому** (после загрузки клиент prepend'ит сверху списка).
3. **`POST /api/channels/:channelId/messages`**:
   - тело: `{ content: string (1..4000), replyToId?: uuid, clientNonce?: string }`
   - `clientNonce` — для дедупликации на клиенте при ретрае. Хранить в `messages.client_nonce` колонке (nullable, индекс unique partial WHERE not null), при повторе с тем же nonce и userId возвращать уже созданное сообщение.
   - Возвращает созданный Message. WS-уведомление участникам канала — отдельно в T-017.
4. **`PATCH /api/messages/:id`** — edit (только автор). Обновляет content, editedAt. 30 дней пост-фактум: после этого PATCH запрещён (защита от ревизионизма).
5. **`DELETE /api/messages/:id`** — soft delete (только автор или admin). Ставит deletedAt и обнуляет content.
6. **Хук на WS из POST/PATCH/DELETE** — отложен в T-017. Сейчас просто структуру эндпоинтов.

## Подсказки
- `uuidv7()` — есть в pg как extension или просто генерируешь на app-уровне (пакет `uuid` 9+). Преимущество перед v4: лексикографически отсортированные = time-sorted, индекс эффективен.
- Soft delete — обнуление content нужно, иначе при exfiltration базы видны удалённые сообщения. Если хочется аудита — оригинал в отдельную таблицу `message_deletions` (только для админов).
- Не используй `LIMIT limit + 1` через interpolation — всегда параметризированно: `LIMIT $1` где `$1 = limit + 1`.
