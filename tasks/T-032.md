# T-032: LiveKit webhooks → WS broadcast

**Phase:** 2 — Голос
**Model:** Opus 4.7 · `high`
**Estimate:** ~2-3 часа

## Контекст
LiveKit может слать webhooks (`participant_joined`, `participant_left`, `room_started`, `room_finished`, `track_published`, `track_unpublished`). Speedy их принимает и broadcast'ит через WS, чтобы UI обновлялся у всех.

Это критично для UX: без этого один user видит участника в комнате через 5-10 секунд (через polling), а должен — мгновенно.

Читать: `ARCHITECTURE.md` §4 (события voice.join/leave/state), LiveKit Webhook docs.

## Файлы в скоупе
- `packages/speedy/src/routes/internal.ts` (новый — internal endpoints, не для клиентов)
- `packages/speedy/src/media/webhook.ts` (новый — обработчик)
- `packages/speedy/src/ws/broadcast.ts` (расширить — broadcast по серверу не только каналу)

## Что сделать
1. **`POST /api/internal/livekit-webhook`**:
   - Заголовок `Authorization` от LiveKit — JWT, подписанный их API key. Валидируем через `livekit-server-sdk` `WebhookReceiver`.
   - Если signature невалидна → 401, никакого fallback.
2. **Обработка событий**:
   - `participant_joined`: парсим room (`voice-${channelId}`), participant identity (= userId), broadcast `{t:'voice.join', channelId, userId}` всем членам сервера канала.
   - `participant_left`: broadcast `{t:'voice.leave', channelId, userId}`.
   - `track_published` (тип screen_share или camera) или `track_unpublished` или мьют изменился: broadcast `{t:'voice.state', channelId, userId, muted, screen}`.
   - `room_finished`: убрать из Redis `voice:channel:<id>:users`.
3. **`broadcast.broadcastToServerMembers(serverId, event)`** — нужно знать всех онлайн-членов сервера. Получаем из БД (`server_members WHERE server_id=...`), отфильтровываем по `connections` Map'у из T-017. Шлём только им.
4. **Idempotency**: LiveKit может слать webhook несколько раз при retry. Используем `event.id` (LiveKit его шлёт) и дедупликацию через Redis SET `livekit:webhook:seen:<id>` с TTL 1 час.
5. **Конфиг LiveKit** (`ops/livekit/livekit.yaml`): уже настроен на webhook URL. Проверь, что URL доступен из контейнера LiveKit (внутри docker-compose это `http://host.docker.internal:3001/api/internal/livekit-webhook` на Mac/Win, на Linux — `http://172.17.0.1:3001/...` или Docker host gateway).

## Out of scope
- Webhooks для recording — recording не делаем.
- Метрики качества (RTT, packet loss) — фаза 5.

## Подсказки
- `host.docker.internal` НЕ работает в Linux Docker без явного `extra_hosts`. В compose добавь:
  ```yaml
  livekit:
    extra_hosts:
      - "host.docker.internal:host-gateway"
  ```
- LiveKit webhook receiver проверяет JWT через тот же API secret. Не путать с auth-токеном клиента — это разные подписи.
- Возможна гонка: webhook participant_joined прилетел до того, как user успел получить ready через WS. Это OK — клиент при ready запрашивает snapshot участников через GET /participants и сам сольёт. Главное — не паниковать в UI.
