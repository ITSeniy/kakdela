# T-001: Проверка и достройка pnpm workspaces

**Phase:** 0 — Фундамент
**Model:** Haiku 4.5 · `low` reasoning
**Estimate:** ~15 минут

## Контекст
- Скаффолд уже содержит `package.json` с workspaces и `pnpm-workspace.yaml` (или поле `workspaces` в корневом package.json).
- Цель — убедиться, что `pnpm install` ставит все 6 пакетов без ошибок.

## Файлы в скоупе
- `package.json` (root, может потребоваться правка)
- `pnpm-workspace.yaml` (создать, если нет)
- `packages/*/package.json` (по одному в каждом пакете)

## Что сделать
1. Создать `pnpm-workspace.yaml` с содержимым:
   ```yaml
   packages:
     - 'packages/*'
   ```
2. Проверить, что у каждого `packages/*/package.json` есть валидное поле `name` со скоупом `@kakdela/*` и `version: "0.0.1"`.
3. Запустить `pnpm install` и убедиться, что нет ошибок и появилась общая `node_modules` + `pnpm-lock.yaml`.
4. Запустить `pnpm -r typecheck` (упадёт, потому что пакеты ещё пустые — это ОК, главное чтобы команда вообще выполнилась).

## Acceptance criteria
- [ ] `pnpm install` завершается без ошибок
- [ ] `pnpm-lock.yaml` создан и закоммичен
- [ ] `pnpm -r run --if-present typecheck` — отрабатывает (даже если кто-то ничего не делает)
- [ ] В каждом пакете есть `name: "@kakdela/<codename>"`

## Out of scope
- Реальная типизация и логика — это T-002…T-005.

## Подсказки
- Если pnpm ругается на peer dependencies — добавь в корневой `.npmrc`:
  ```
  auto-install-peers=true
  strict-peer-dependencies=false
  ```
