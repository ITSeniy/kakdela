# T-050a: Захват системного звука — проверка на Win10/11

**Phase:** 3 — Демонстрация экрана ⭐ MVP
**Model:** Opus 4.7 · `high` (рискованное место для Tauri/WebView2)
**Estimate:** ~2-3 часа (включая ручное тестирование)

## Контекст
`getDisplayMedia({ audio: true })` в Chromium на Windows захватывает системный звук (с экрана/окна источника) — это **критично** для демонстрации игры с её звуком, видео с YouTube и т.п. Но в WebView2 поведение может отличаться.

Это **research task** + fallback логика.

1. Собрать `pnpm tauri dev` на Windows 11 (не WSL!), запустить.

### Шаг 3: capability detection + fallback

В `useScreenShare`:

```ts
async function tryStartShare() {
  try {
    await room.localParticipant.setScreenShareEnabled(true, { audio: true })
    store.setAudioCaptureSupported(true)
  } catch (err) {
    // Если ошибка указывает на проблему с audio constraint (по сообщению или коду),
    // retry без audio:
    if (isAudioConstraintError(err)) {
      await room.localParticipant.setScreenShareEnabled(true, { audio: false })
      store.setAudioCaptureSupported(false)
      toast('Системный звук не захвачен — браузер не поддерживает на этой платформе')
    } else {
      throw err
    }
  }
}
```

### Шаг 4: UI

В VoiceControls, рядом с кнопкой «демо», маленький toggle «со звуком» (вкл по умолчанию). Если `audioCaptureSupported === false` — toggle disabled с tooltip'ом «на вашей системе недоступно».

## Подсказки
- Если на Win11 не работает — попробовать поднять WebView2 Runtime до последней (через Microsoft Edge Update). WebView2 связан с Edge, не с Chrome.
- LiveKit имеет `screenShareAudioEnabled` опцию — она просто прокси над `audio: true` в getDisplayMedia.
- Если упрётся — это и есть **тот случай**, когда стоит подумать о миграции на Electron. Но сначала исчерпать варианты с WebView2.
