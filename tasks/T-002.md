# T-002: docker-compose поднимается на чистой машине

**Phase:** 0 — Фундамент
**Model:** Haiku 4.5 · `low` reasoning (это ручная проверка)
**Estimate:** ~20 минут

## Контекст
- `docker-compose.dev.yml` и `ops/livekit/livekit.yaml` уже в скаффолде.
- Цель — пройти полный путь поднятия инфры и убедиться, что всё реально стартует.

## Что сделать
1. Скопировать `.env.example` → `.env`, сгенерировать `JWT_*_SECRET` через `openssl rand -hex 64`.
2. `pnpm docker:dev` (или `docker compose -f docker-compose.dev.yml up`).
3. Подождать «MinIO bucket ready» в логах `kd-minio-init`.
4. Проверить руками:
   - `psql postgres://kakdela:kakdela_dev@localhost:5432/kakdela -c '\dt'` (откроет пустую базу).
   - `redis-cli -h localhost PING` → `PONG`.
   - `http://localhost:9001` → MinIO web (логин `kakdela` / пароль из .env).
   - `http://localhost:7880` → LiveKit HTTP statuspage (или хотя бы 200 на `/`).
5. Сделать `docker compose down`, потом снова `up` — данные должны сохраниться.
6. Сделать `pnpm docker:reset` — volumes должны удалиться.

## Acceptance criteria
- [ ] Все 4 контейнера в статусе `healthy` / `running` (`docker ps`).
- [ ] Bucket `kakdela` существует в MinIO.
- [ ] LiveKit логирует «starting LiveKit server» без ошибок.
- [ ] Документировать в README, если на твоей машине понадобился какой-то workaround (например, `network_mode: host` для LiveKit на Linux).

## Out of scope
- Никаких изменений в инфре — только проверка и фикс конфига, если что-то реально не работает.

## Подсказки
- Если LiveKit ругается на UDP-порты — на Linux попробуй `network_mode: host` (закомментировать `ports:`).
- На Windows в WSL2 контейнерах UDP-проброс обычно работает «как есть», но если будут проблемы — переключи Docker на «WSL 2 backend».
