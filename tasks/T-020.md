# T-020: Presence — online/offline через Redis

**Phase:** 1 — Текстовый дом
**Model:** Sonnet 4.6 · `medium`
**Estimate:** ~1.5-2 часа

## Контекст
Статус пользователя (online / idle / dnd / offline) определяется WS-соединениями. Redis служит для шаринга между процессами speedy и для быстрого ответа `/members`.

## Файлы в скоупе
- `packages/speedy/src/presence/store.ts` (новый — Redis API)
- `packages/speedy/src/ws/server.ts` (хуки connect/disconnect)
- `packages/speedy/src/routes/servers.ts` (использовать presence в /members)

## Что сделать
1. **Схема в Redis**:
   - Ключи: `presence:user:<userId>` → hash `{ status, customStatus, lastSeen, connectionCount }`
   - TTL не ставим — обновляем явно.
2. **`PresenceStore`**:
   - `addConnection(userId)` → INCR connectionCount, если был 0 → SET status='online', publish presence-event.
   - `removeConnection(userId)` → DECR. Если стал 0 → SET status='offline' (с задержкой 30s через debounce, см. подсказку).
   - `getStatus(userId)` → читает hash.
   - `getStatusBulk(userIds)` → MGET через pipeline, для /members.
   - `setCustomStatus(userId, text)` — для /me PATCH (в фазе 5, пока заглушка).
3. **WS-хуки**:
   - При успешном hello + ready → `presence.addConnection(userId)` → broadcast `{t:'presence', userId, status:'online'}` всем серверам user'а.
   - При close → `removeConnection`, через 30s проверка: если всё ещё 0 → broadcast offline.
4. **`/api/servers/:id/members`** (расширить из T-015): добавить `status` через `getStatusBulk`. Не делать N+1 SELECT, batch.
5. **Idle detection** (опционально для MVP): клиент шлёт WS-событие `{t:'presence', status:'idle'}` после 5 минут без активности (mousedown/keydown). Сервер сохраняет, broadcast'ит. При возврате — обратно 'online'.

## Out of scope
- DND/manual idle через UI — фаза 5 (вместе с настройками).
- Last seen «5 минут назад» в UI — фаза 5.

## Подсказки
- Debounce offline на 30s важен: при reconnect (потеря wifi на 5s) не хочется моргания online → offline → online. Реализация: SET ключ `presence:disconnect-pending:<userId>` с TTL 30s; при connect — DEL; при истечении (через keyspace notifications или периодический cron в speedy) — выставляем offline.
- Альтернатива проще: после remove проверить через `setTimeout(30s)` connection count. Это работает в одном процессе, но не в multi-instance. Для MVP — норм.
- Не используй Redis pub/sub для presence broadcast — используй ту же broker-абстракцию из T-017.
