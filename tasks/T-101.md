# T-101: Крипто-ядро — libsignal в Rust + ключи (device-bound)

**Phase:** 6 — Мобильный клиент / Секретные чаты
**Model:** Opus 4.7 · `high` — **SECURITY, крипто. Не импровизировать.**
**Estimate:** ~6-8 часов

## Контекст

Сердце E2EE. `libsignal` (ядро на Rust) линкуется в `src-tauri` и работает на нативной стороне — WebView ключей **не видит**. Протокол: **X3DH** для установки сессии + **Double Ratchet** для сообщений (forward secrecy из коробки). Это стандартный, проверенный 1:1-протокол Signal/WhatsApp/секретных чатов Telegram — мы его **используем**, а не переизобретаем.

**Device-bound** (решение фазы 6): identity-ключ генерируется на устройстве, приватная часть запечатана Android Keystore, **никогда** не покидает телефон и **не синхронизируется**. Потеря устройства = потеря секретных переписок. Это by design.

Сервер выступает **слепым каталогом ключей** (key directory): хранит только **публичные** prekey-бандлы, не может расшифровать ничего. Это единственная серверная часть этой карточки.

Читать: `.claude/CONVENTIONS.md` (Безопасность; «нет секретов в логах»), `tasks/T-100.md` (host-layer), документацию libsignal (X3DH, Double Ratchet, prekey bundle).

## Что сделать

### Нативная сторона (Rust, `src-tauri`)

1. **Identity**: при первом запуске — сгенерировать identity keypair (по схеме libsignal) + `registrationId`. Приватный ключ хранить запечатанным: в Keystore лежит **wrapping-ключ** (StrongBox если доступен), им шифруется локальный стор (Keystore не хранит произвольные байты — только ключ-обёртку).
2. **Prekeys**: сгенерировать signed prekey + N (например 100) one-time prekeys. Наружу отдаются **только публичные** части.
3. **Стор** (`store.rs`): реализовать `IdentityKeyStore` / `SessionStore` / `PreKeyStore` / `SignedPreKeyStore` libsignal поверх зашифрованного локального хранилища. Источник истины сессий — устройство.
4. **Команды** (`commands.rs` → JS):
   - `crypto_init()` — идемпотентно создаёт identity при первом вызове.
   - `crypto_publish_keys()` — формирует публичный бандл для отправки на сервер.
   - `crypto_encrypt(toUserId, plaintext) -> { ciphertext, msgType }` — X3DH при первом сообщении, дальше Double Ratchet.
   - `crypto_decrypt(fromUserId, ciphertext, msgType) -> plaintext`.
   - `crypto_session_exists(userId) -> bool`.
   - `crypto_safety_number(userId) -> string` — детерминированный, **симметричный** (одинаковый у обеих сторон).

### Серверная сторона (слепой каталог, `speedy`)

5. **Таблицы**: `identity_keys (userId pk, identityPubKey, registrationId)`, `signed_prekeys (userId, keyId, pubKey, signature)`, `one_time_prekeys (userId, keyId, pubKey, consumedAt)`. **Только публичные ключи. Приватных в БД нет и быть не может.**
6. **Эндпоинты** (`keys.ts`, все authenticated, Zod на каждом):
   - `POST /api/keys/bundle` — загрузить свой публичный бандл (identity + signed prekey + пачка one-time).
   - `GET /api/keys/:userId/bundle` — получить бандл для старта сессии; сервер **отдаёт и помечает consumed** один one-time prekey (atomic).
   - `POST /api/keys/topup` — долить one-time prekeys, когда осталось мало.
   - `GET /api/keys/count` — сколько one-time осталось (для триггера topup на клиенте).

### Host-абстракция

7. `host/crypto.ts` оборачивает команды; на web-пути (`pnpm dev:web`) бросает `{ code: 'secret-chats-unsupported' }` — секретные чаты только на мобиле.

## DoD

- [ ] Чистая установка генерирует identity + бандл, на сервере появляются **только публичные** ключи
- [ ] `crypto_encrypt` → `crypto_decrypt` round-trip строки между двумя эмуляторами (два разных user'а)
- [ ] Перехват трафика (прокси/логи): приватные ключи **ни в одном** запросе не появляются
- [ ] `GET /api/keys/:userId/bundle` атомарно консьюмит один one-time prekey; `topup` восполняет
- [ ] `crypto_safety_number` детерминирован и одинаков у обеих сторон сессии
- [ ] В логах сервера и клиента **нет** секретного материала (CONVENTIONS)
- [ ] `pnpm typecheck` + `cargo check` (android target) чистые

## Out of scope

- Групповые сессии (MLS) — нет, мобилка 1:1.
- Мульти-девайс / синк ключей между телефоном и desktop — нет (device-bound).
- Бэкап/восстановление ключей — нет (device-bound = потеря устройства = потеря истории, by design).
- Post-quantum, sealed sender — нет.
- Сам обмен сообщениями (envelope queue) — это T-102. Здесь только крипто-примитивы + каталог ключей.

## Подсказки

- Rust-ядро libsignal **не публикуется** как стабильный crate — пин на конкретный git-rev либо vendoring; обновлять осознанно.
- Сборка под Android — `cargo-ndk` (см. `docs/MOBILE.md` из T-100).
- **Не писать свою крипту.** Использовать session-API libsignal как есть. Любое «оптимизирую сам» в этой карточке — красный флаг.
- Keystore хранит маленький wrapping-ключ, а не сессии целиком; сессии — в зашифрованном локальном сторе.
- `registrationId` и id prekey'ев — целые; следить за уникальностью one-time keyId.
- Аккуратно с трейсингом: ни `dbg!`, ни `tracing` не должны проглатывать приватные байты.
