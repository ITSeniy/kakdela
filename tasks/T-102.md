## Контекст

Секретным чатам нужен транспорт, который сервер **не может прочитать**, плюс **локальная история на устройстве** (device-bound → сервер не хранит плейнтекст, история живёт только на телефоне). Сервер работает как **store-and-forward очередь шифр-конвертов**, удаляя их после доставки (модель Signal). Это **параллельный** путь к обычным cloud-DM из T-064 — обычные DM не трогаем.

Завязано на T-101 (crypto_encrypt/decrypt) и архитектурное правило: **все мутации через REST, WebSocket — только receive-only** уведомление «тебе письмо».

## Что сделать

### Сервер — слепая очередь (`speedy`)

1. **Таблица `secret_envelopes`**: `id uuid pk, fromUserId fk, toUserId fk, ciphertext bytea, msgType smallint, createdAt`. **Нет колонки content/text.** Индекс `(toUserId, id)`. `ciphertext` — `bytea`, не `text` (его нельзя гонять через tsvector).
2. **Эндпоинты** (`secret-chat.ts`, authenticated, Zod):
   - `POST /api/secret/send` — `{ toUserId, ciphertext (base64), msgType }`. Сервер кладёт **непрозрачный** blob, **не парсит**. Если получатель онлайн — шлёт WS `secret.envelope`.
   - `GET /api/secret/inbox` — pending-конверты для меня.
   - `POST /api/secret/ack` — `{ ids: [] }` — удалить доставленные. После ack blob физически удаляется.
   - **Auth**: отправитель должен иметь право писать получателю — переиспользовать логику доступа из T-064 (общий сервер / разрешённый DM).
3. **Retention**: cron чистит недоставленные конверты старше 30 дней (offline-получатель). Зафиксировать cap в конфиге.
4. **WS** (`secret.envelope`): несёт **только** «есть письмо» + `id`/`fromUserId`, **никакого контента**. Receive-only, как вся WS-архитектура.

### Устройство — локальная история (Rust, `src-tauri`)

5. **Зашифрованная SQLite** (`local_db.rs`): ключ шифрования — wrapping-ключ из T-101 (Keystore). Схема: `messages (id, peerUserId, direction, plaintext, msgType, sentAt, readAt, status)`. Это **источник истины** истории секретных чатов.
6. **Device-bound по-настоящему**: при wipe данных приложения / переустановке локальная БД и ключи исчезают → история секретных чатов теряется. Это ожидаемое поведение, не баг.

### Pipeline (`features/secret/api.ts` + `host/secret-store.ts`)

7. **Отправка**: plaintext → `host.crypto.encrypt(toUserId)` → `POST /api/secret/send` → записать свою копию в локальную БД (`status='sent'`) → инвалидировать query.
8. **Приём**: WS `secret.envelope` → `GET /api/secret/inbox` → по каждому `host.crypto.decrypt(fromUserId)` → запись в локальную БД → `POST /api/secret/ack` → инвалидировать query.
9. **Контрол-сообщения**: read-receipt и typing — это **типизированные конверты** (`msgType='read'` / `'typing'`), идущие тем же E2EE-каналом, **не** отдельным REST. Прочитанность (`✓✓`) ставится по входящему `read`-конверту.

## DoD

- [ ] A→B (оба онлайн): сообщение шифруется, доходит, расшифровывается у B; серверная строка удаляется после ack
- [ ] B оффлайн → конверт лежит в очереди → доставляется при reconnect
- [ ] Инспекция серверной БД показывает **только `ciphertext bytea`**, ноль плейнтекста
- [ ] Локальная история переживает рестарт приложения (читается из зашифрованной SQLite)
- [ ] Переустановка / wipe данных → секретная история исчезла (device-bound подтверждён)
- [ ] WS `secret.envelope` не содержит контента (проверить payload)
- [ ] Read-receipt `✓✓` ставится через входящий control-конверт
- [ ] `pnpm typecheck` + `cargo check` чистые

## Out of scope

- UI секретного чата — T-103. Здесь pipeline + хранилища.
- Самоуничтожающиеся сообщения (таймер) — отдельная задача.
- Вложения/картинки в секретных чатах — пока **только текст**.
- Серверный поиск по секретным чатам — **невозможен by design**, не делаем.
- Мульти-девайс синк, бэкап истории — нет (device-bound).

## Подсказки

- `ciphertext` строго `bytea`, не `text` — иначе кто-нибудь случайно повесит на него GIN/tsvector.
- Обычный DM из T-064 **не трогаем** — секретный чат это параллельная ветка, отдельные таблицы и роуты.
- WS остаётся «глупым звоночком»: контент только в зашифрованном конверте по REST, в WS — лишь факт.
- Зашифрованная локальная БД: `tauri-plugin-sql` + SQLCipher, либо app-private storage. Ключ — из T-101.
- Retention-cron не должен трогать доставленные (их уже нет) — только висящие offline-конверты.
