# T-091: Search — фильтры channel / author / date

**Phase:** 5 — Полировка
**Model:** Sonnet 4.6 · `medium`
**Estimate:** ~1.5-2 часа

## Контекст

`SearchScreen.tsx` сейчас показывает только input + сортировку. Backend `/api/search/messages` (T-066) уже принимает `channelId?, authorId?, before?, after?` — функционал полностью готов, нужен UI.

Читать: `tasks/T-066.md`, `packages/speedy/src/routes/search.ts`.

## Файлы в скоупе

- `packages/polly/src/features/search/SearchScreen.tsx` (доработка)
- `packages/polly/src/features/search/SearchFilters.tsx` (новый — sidebar с фильтрами)
- `packages/polly/src/features/search/api.ts` (передать новые параметры)

## Цель

Пользователь может сузить выдачу поиска по каналу, автору и диапазону дат.

## Что сделать

1. **`SearchFilters.tsx`** — sidebar 240px слева от результатов:
   - **«В канале»**: select со списком всех доступных user'у каналов (через `getServersForUser` + flat list `#channel-name · сервер`). Опция «везде» (default).
   - **«От»**: select из member'ов всех общих серверов. Опция «любой». Поисковая строка вверху списка — длинный список 50+ людей надо фильтровать.
   - **«Когда»**: три radio — «всё время» / «за неделю» / «за месяц» / «свой диапазон». Если свой — два `<input type="date">`.
   - Кнопка «сбросить фильтры» внизу, активна если что-то выбрано.
2. **State**: фильтры локально в `SearchScreen` через `useState`:
   ```ts
   const [filters, setFilters] = useState<{
     channelId: string | null
     authorId: string | null
     period: 'all' | 'week' | 'month' | 'custom'
     before: string | null
     after: string | null
   }>(...)
   ```
3. **Передача в API**: расширить `searchMessages` в `api.ts` — добавить параметры. При смене фильтров — `useQuery` с queryKey, включающим filters.
4. **Layout**:
   - 1024px+: filters слева 240px, results 1fr — двухколоночный grid.
   - <1024px: filters collapsible через кнопку «фильтры» в шапке.
5. **URL-state** (опционально): сериализовать фильтры в `?q=...&channel=...&author=...&before=...` — чтобы можно было поделиться ссылкой на конкретный search. Использовать `useLocation` от wouter + `URLSearchParams`.

## Acceptance criteria (DoD)

- [ ] Выбрать канал «#общее» → результаты только оттуда
- [ ] Выбрать автора «Вася» → только сообщения Васи
- [ ] «За неделю» → автоматически `after = now - 7d`, в URL и в API
- [ ] «Сбросить» → возвращается ко «всё время / любой канал / любой автор»
- [ ] На узком окне фильтры сворачиваются в collapsible
- [ ] Если выбрать канал чужого сервера через URL-state — backend вернёт 403, UI показывает «нет доступа к каналу»
- [ ] `pnpm typecheck` чистый

## Out of scope

- Фильтр по типу attachment (только картинки / только файлы) — отдельная задача если попросят.
- Сохранение «избранных» поисков — нет.
- Полнотекстовый поиск по attachment'ам — не поддерживается postgres FTS из коробки.

## Подсказки

- При смене фильтров queryKey меняется → TanStack Query автоматически делает новый запрос с debounce из существующего DEBOUNCE_MS. Не пиши свой debounce поверх.
- Date pickers: native `<input type="date">` хорошо работает в Chromium (WebView2 это Chromium) — не тащи react-datepicker.
- Member-список для «от»: использовать общий `useMembersAcrossServers` хелпер если его уже нет — нужен будет и для @mentions глобально.
