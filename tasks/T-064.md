# T-064: DM — личные сообщения

**Phase:** 4 — Уют
**Model:** Sonnet 4.6 · `medium`
**Estimate:** ~2-3 часа

## Контекст
Пары пользователей как «канал» kind='dm'. На 15-20 человек это N*(N-1)/2 = ~190 потенциальных DM, но реально создаются лениво при первом сообщении. Дизайн в `designs/final-dm.jsx`.

## Файлы в скоупе
- `packages/speedy/src/db/schema.ts` (дополнить — `dm_pairs` или просто использовать channels с kind='dm')
- `packages/speedy/src/routes/dm.ts` (новый)
- `packages/polly/src/features/dm/DmList.tsx` (новый — список открытых DM)
- `packages/polly/src/features/dm/DmScreen.tsx` (новый)
- `packages/polly/src/features/dm/api.ts` (новый)

## Что сделать
1. **Модель**:
   - Используем существующую `channels` таблицу с `kind='dm'`, но `serverId` — nullable для DM.
   - Новая таблица `channel_members` (для DM указывает кто в нём; для server-каналов не нужна).
   - Альтернатива: отдельная `dm_channels (id, userAId, userBId)` — проще, без `serverId` зайки.
   - **Выбор**: `dm_channels` — проще и яснее. Использовать.
2. **Эндпоинты**:
   - `GET /api/dm` → список DM текущего user'а: `[{ channelId, otherUser, lastMessage, unreadCount }]`.
   - `POST /api/dm/with/:userId` → возвращает existing или создаёт новый DM. Идемпотентно.
   - Messages reused — те же `POST /api/channels/:id/messages` (теперь auth-проверка должна понимать DM: user должен быть один из двух participants).
3. **Permission helper**: `assertCanAccessChannel(userId, channelId)`:
   - Если channel.kind === 'dm' → check `channel.userAId === userId || channel.userBId === userId`
   - Если 'text'/'voice' → check membership в server.
4. **DmList в шелле**:
   - Когда user кликает «дом» в server rail (T-018, заглушка) — переключается на DM view.
   - Левая колонка: список DM (`<DmList>`), сортировка по lastMessageAt.
   - Карточка: avatar + displayName + preview последнего сообщения + unread badge.
5. **DmScreen**:
   - По сути то же, что ChatScreen (T-019), только header показывает имя собеседника, нет topic'а / member list'а.
   - Можно переиспользовать большую часть кода ChatScreen, оборачивая в новый layout.
6. **Открытие DM из других мест**:
   - Клик по member в server members list (T-068 profile modal) → кнопка «написать сообщение» → POST /api/dm/with/:id → перейти на /dm/:userId.
   - URL pattern: `/dm/:channelId` или `/dm/with/:userId` (последнее удобнее для ссылок).

## DoD
- [ ] Клик «написать сообщение» на профиле другого user'а → открывает DM
- [ ] DM list показывает все начатые DM, отсортированные по последнему сообщению
- [ ] Unread badge корректный
- [ ] Сообщения отправляются/приходят через тот же WS, что обычный чат
- [ ] Невозможно открыть чужой DM через URL — 403
- [ ] `pnpm typecheck` чистый

## Out of scope
- Групповые DM (3+ человек) — нет, для этого есть serverside каналы.
- E2E шифрование DM — нет, доверяем self-host'у.

## Подсказки
- WS subscribe для DM: при ready клиент подписывается также на DM-channels через `subscribedChannels` (T-017). Backend знает все DM user'а через `dm_channels`, добавляет их в subscribe set автоматически.
- Не делай Unread через JOIN с messages при каждом запросе — храни `last_read_message_id` в `dm_channels` для каждой стороны.
