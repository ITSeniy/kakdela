# T-010: Auth — register / login / refresh / me

**Phase:** 1 — Текстовый дом
**Model:** Opus 4.7 · `high` reasoning

## Контекст
Безопасность — единственная вещь, которую нельзя «починить потом». Здесь важно: argon2id (НЕ bcrypt), JWT access + refresh с ротацией, защита от тайминг-атак на login, rate limiting.

Читать: `ARCHITECTURE.md` §5 (data model — таблицы users, sessions), §6 (security baseline), `.claude/CONVENTIONS.md` (раздел «Безопасность»).

## Что сделать
1. **Таблица `sessions`** в schema.ts:
   - `id` uuid pk
   - `userId` fk → users.id
   - `refreshTokenHash` text (sha256 от refresh-токена, никогда не plain)
   - `userAgent` text
   - `ipAddress` text
   - `createdAt`, `lastUsedAt`, `expiresAt` timestamps
   - индекс по `userId`
2. **passwords.ts**: `hash(password)` и `verify(password, hash)` через `@node-rs/argon2` с настройками `{ memoryCost: 19456, timeCost: 2, parallelism: 1 }` (OWASP-рекомендуемые).
3. **tokens.ts**: `issueAccessToken(userId)` (15m), `issueRefreshToken(userId)` (30d, сохраняется hash в `sessions`), `verifyAccessToken(token)`. Использовать `jose` для HS256.
4. **Эндпоинты**:
   - `POST /api/auth/register` — принимает `{ inviteCode, username, displayName, email, password }`, валидирует инвайт (см. T-011), создаёт user, возвращает токены.
   - `POST /api/auth/login` — `{ email, password }`, защита от тайминга (всегда выполнять argon2 verify, даже если user не найден, с фейк-хешем).
   - `POST /api/auth/refresh` — `{ refreshToken }`, ротация: старый refresh инвалидируется, выдаются новые access + refresh.
   - `POST /api/auth/logout` — удаляет текущую сессию.
   - `GET /api/auth/me` — требует `requireAuth`, возвращает текущего пользователя.
5. **Rate limiting** на `/login` и `/register`: 10 попыток с одного IP в минуту, через `@fastify/rate-limit` + redis store.
6. **Куки vs заголовок**: refresh-токен в httpOnly cookie с `SameSite=Strict`, access-токен — в JSON body (клиент держит его в памяти/stronghold). Это компромисс: httpOnly защищает от XSS, в-памяти — от чтения чужих скриптов.

## DoD
- [ ] `curl -X POST .../register` создаёт пользователя, возвращает `{ accessToken, user }` + ставит refresh cookie
- [ ] `curl -X POST .../login` для несуществующего email отвечает за то же время, что для существующего с неверным паролем (тайминг-тест — `time` в bash должен показывать разброс < 50ms)
- [ ] Невалидный JWT → 401, истёкший → 401 с кодом `token-expired`
- [ ] Refresh ротирует: старый refresh после использования не работает
- [ ] Rate limit реально работает: 11-й запрос за минуту → 429
- [ ] `pnpm typecheck` чистый

## Out of scope
- Восстановление пароля по email (нет SMTP в MVP).
- 2FA.
- OAuth (нет внешних провайдеров).

## Подсказки
- argon2id, НЕ argon2i и НЕ argon2d. Пакет `@node-rs/argon2` имеет вариант по умолчанию — проверить, что это id.
- JWT секреты должны быть **разными** для access и refresh, иначе можно использовать refresh как access (что разрушает refresh-логику).
- Не клади JWT в localStorage клиента — это XSS-катастрофа. На клиенте access живёт в Zustand-store без persist, refresh — в stronghold.
- Fake-hash для timing-protection: предгенерированный argon2id-хеш от какой-нибудь длинной строки, заранее загруженный в память.
