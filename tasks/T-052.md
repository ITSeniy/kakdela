# T-052: Quality presets (auto / 1080p30 / 720p30)

**Phase:** 3 — Демонстрация экрана ⭐ MVP
**Model:** Sonnet 4.6 · `medium`
**Estimate:** ~1-1.5 часа

## Контекст
LiveKit publishes screen tracks с дефолтными constraints (1080p, ~3 Mbps). Это многовато для VPS с 2 vCPU на 5+ одновременных трансляциях. Даём user выбор.

## Файлы в скоупе
- `packages/polly/src/features/voice/store.ts` (новое поле — `screenQuality`)
- `packages/polly/src/features/voice/useScreenShare.ts` (применять preset)
- `packages/polly/src/features/voice/VoiceControls.tsx` (UI выбора)
- `packages/polly/src/features/settings/VoiceSettings.tsx` (дублирование в настройках)

## Что сделать
1. **Presets** (типизировать в TS):
   ```
   const QUALITY_PRESETS = {
     auto:    { /* let LiveKit decide */ },
     '1080p30': { resolution: { width: 1920, height: 1080, frameRate: 30 }, bitrate: 3_000_000 },
     '720p30':  { resolution: { width: 1280, height: 720,  frameRate: 30 }, bitrate: 1_500_000 },
     '720p15':  { resolution: { width: 1280, height: 720,  frameRate: 15 }, bitrate: 800_000 },
   }
   ```
2. **При startShare** — применить preset через `ScreenShareCaptureOptions` LiveKit'а:
   ```
   await room.localParticipant.setScreenShareEnabled(true, {
     audio: store.audioCaptureSupported,
     ...QUALITY_PRESETS[store.screenQuality],
   })
   ```
3. **UI**:
   - В `VoiceControls` рядом с кнопкой «демо» — маленький dropdown «качество». При смене во время трансляции — restart share (unpublish + publish с новыми params).
   - В будущем `VoiceSettings` (для T-068) — то же самое, как persistent setting.
4. **Adaptive 'auto'**:
   - LiveKit имеет `simulcast` для screen share (но не во всех версиях SDK). Включи `videoSimulcastLayers` если поддерживается — это даёт SFU возможность даунгрейдить качество для слабых клиентов автоматически.
   - В preset `auto` — задействуем simulcast.

## DoD
- [ ] Переключение качества на лету → screen share продолжается с новыми параметрами
- [ ] На 720p15 битрейт реально ниже (проверить через `chrome://webrtc-internals` или LiveKit stats)
- [ ] Выбор сохраняется между сессиями (localStorage)
- [ ] `pnpm typecheck` чистый

## Out of scope
- Per-receiver bandwidth control (отдельный качество для каждого зрителя) — нет.
- Hardware encoding (NVENC, etc.) — браузеру не управляется руками.

## Подсказки
- Restart share при смене quality: `await stopShare(); await startShare()`. Между ними — 200-500ms задержка (LiveKit нужно опубликовать новые tracks).
- Битрейт 3 Mbps × 5 человек = 15 Mbps upstream только для screens. На домашнем интернете 50 Mbps upload — впритык. Дефолт лучше '720p30'.
