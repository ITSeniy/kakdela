# T-094: Нативный захват звука для демонстрации (Windows/WASAPI)

**Phase:** 5 — Полировка (отдельный нативный заход)
**Model:** Opus 4.7 · `high` (нативный realtime-аудио, COM/WASAPI, мост в WebRTC)
**Estimate:** Stage 0 — готов; A/B/C — несколько сессий с ручным прогоном на железе

## Контекст

«Дискордовский» захват звука: дать в демонстрацию экрана звук **конкретного приложения**, а не только весь системный. Решено НЕ мигрировать на Mediasoup и НЕ писать свой SFU — транспорт остаётся LiveKit, нативно захваченный PCM публикуется кастомным треком в существующую комнату (ARCHITECTURE.md §3.5).

Фича разваливается на ступени по доступности API в Windows:
- **System loopback** (весь звук устройства вывода, `AUDCLNT_STREAMFLAGS_LOOPBACK`) — Vista+, гейта нет.
- **Process loopback** (звук одного приложения, `ActivateAudioInterfaceAsync` + `AUDIOCLIENT_PROCESS_LOOPBACK_PARAMS`) — API с **Windows 10 version 2004 = build 19041** (май 2020). Microsoft в сэмпле документирует минимум build 20348 (Server 2022/Win11), но на практике (OBS win-capture-audio, Discord) работает с 19041. Поэтому порог = **19041** (не Win11-only), + runtime-фолбэк.

Источники по билду: [ApplicationLoopback sample](https://learn.microsoft.com/en-us/samples/microsoft/windows-classic-samples/applicationloopbackaudio-sample/), [AUDIOCLIENT_PROCESS_LOOPBACK_PARAMS](https://learn.microsoft.com/en-us/windows/win32/api/audioclientactivationparams/ns-audioclientactivationparams-audioclient_process_loopback_params), [bozbez/win-capture-audio](https://github.com/bozbez/win-capture-audio).

## Стадии

### Stage 0 — определение возможностей ОС ✅ ГОТОВ
- Rust: `detect_capability()` через `RtlGetVersion` (ntdll) → `{ mode, systemLoopback, processLoopback, buildNumber }`. Порог `PROCESS_LOOPBACK_MIN_BUILD = 19041`.
- Команда `audio_capture_capability`, host-обёртка + хук, read-only строка в VoiceSettings.
- `cargo check` чистый, `pnpm --filter @kakdela/polly typecheck` чистый.

### Stage A — system loopback capture (весь системный звук) ✅ КОД ГОТОВ (ждёт прогона)
- `src-tauri/src/audio/capture.rs`: WASAPI loopback на default render endpoint (`AUDCLNT_STREAMFLAGS_LOOPBACK`) через `windows` crate напрямую (не обёртку — сырой API однозначен и полностью compile-checked; `wasapi` 0.23 имеет волатильный API и неясный loopback-флаг).
- Поллинг ~10 мс (без событийной модели — при тишине события не приходят), нормализация любого микс-формата (float32/64, PCM int16/24/32) → 16-бит PCM, потоковый WAV с патчем размеров в finalize.
- **Verification-артефакт:** команда `audio_capture_record(seconds)` пишет WAV в temp и возвращает путь; dev-кнопка в VoiceSettings («тест: записать 5 сек», только `import.meta.env.DEV`). Проигрываешь WAV — слышишь системный звук.
- Без билд-гейта (работает на всём Win10/11). `cargo check` + `pnpm typecheck` чистые.
- Каверзы, которые проверяются ТОЛЬКО на железе: реальный микс-формат (ожидаем float32 48k stereo), отсутствие хрипов/пропусков, поведение при полной тишине, ресемплинг к 48k (НЕ здесь — Stage C).

### Stage B — process loopback (звук приложения) ✅ КОД ГОТОВ (ждёт прогона)
- `activate_process_loopback_client`: `ActivateAudioInterfaceAsync(VIRTUAL_AUDIO_DEVICE_PROCESS_LOOPBACK)` с `AUDIOCLIENT_PROCESS_LOOPBACK_PARAMS` (INCLUDE_TARGET_PROCESS_TREE — захват PID + дочерних, важно для браузеров: аудио в child-процессе). Async COM: completion handler через `#[implement(IActivateAudioInterfaceCompletionHandler)]` сигналит event, `GetActivateResult` отдаёт IAudioClient. Параметры упакованы в `PROPVARIANT`/VT_BLOB.
- Формат задаём сами (48к/16/стерео + `AUDCLNT_STREAMFLAGS_AUTOCONVERTPCM` — у виртуального устройства нет endpoint-микса). Event-driven цикл (как в MS ApplicationLoopback), дренаж/WAV — общие с Stage A.
- Пикер: `audio_list_processes` (toolhelp snapshot — PID + exe). Сейчас ВСЕ процессы; фильтр «только звучащие» (IAudioSessionManager2) — возможный B2.
- Команда `audio_capture_record_process(pid, seconds)` + dev-пикер в VoiceSettings (только при `processLoopback`). `cargo check` + typecheck чистые.
- **Runtime-фолбэк на Stage A** при ошибке активации пока НЕ автоматический (debug-команда просто отдаёт ошибку) — фолбэк-оркестрация логически принадлежит Stage C (где захват идёт в реальную демку).
- Каверзы, только на железе: реально ли event-driven отдаёт пакеты для process-loopback (если нет — переключить на polling), захват дерева браузера, поведение при тишине процесса.

### Stage C — мост PCM → LiveKit. **Выбран C1 (WebView)** — обсуждение 2026-06-30
Решение: НЕ C2 (LiveKit Rust SDK). C2 = второе подключение + фантомный участник против identity-логики `lib/livekit.ts` + libwebrtc bloat (~×4-5 инсталлятор, против ценности «~10МБ» из ARCHITECTURE §3.5). Латентность C2 для односторонней демки несущественна. MSTG в WebView2 подтверждён (`mstg:true, audioData:true`).

**Шаг 1 — транспорт Rust→JS ✅ КОД ГОТОВ (ждёт прогона)**
- `stream_capture(pid?, stop, sink)`: system (polling) и process (event-driven), оба к фикс-формату 48к/16/стерео через AUTOCONVERTPCM (JS знает формат, ресемплинг не нужен — 48к для Opus). Stop-флаг + поток.
- `audio_stream_start(pid?, Channel)` / `audio_stream_stop` + managed `AudioStreamState`. PCM идёт **сырыми байтами** через `InvokeResponseBody::Raw` → `ArrayBuffer` в JS, без re-encode (tauri 2.11 это поддерживает; проверено в src).
- host `startAudioStream(opts, onPcm)` + dev-кнопка «тест стрима PCM (3 сек)» (считает полученные КБ vs ожидание ~562 КБ). WAV-debug команды Stage A/B НЕ тронуты.
- `cargo check` + typecheck чистые.

**Шаг 2 — JS-сink (MSTG) ✅ КОД ГОТОВ (ждёт прогона)**
- `features/voice/nativeAudioTrack.ts`: `createNativeAudioTrack({pid?})` → `ArrayBuffer` чанки → `AudioData` (format `s16`, 48к/2ch, монотонный timestamp) → `MediaStreamTrackGenerator` → живой `MediaStreamTrack`. Не ждём `write` (внутренняя очередь writable сглаживает); reject = трек закрыт → стоп.
- `MediaStreamTrackGenerator` нет в lib.dom → минимальный ambient-декл `src/types/dom-mstg.d.ts` (`AudioData` уже есть в lib.dom).
- Dev-проверка: кнопка «слушать через MSTG» в пикере процесса — захват выбранного app → проигрывание через полный пайплайн в `<audio>`. Фидбека нет (ловим процесс, не системный вывод). `typecheck` чистый.
- Явный jitter-буфер пока не добавлял — полагаемся на очередь writable + timestamp; добавим, если на железе будут глитчи.

**Шаг 3 — интеграция в `useScreenShare` ✅ КОД ГОТОВ (ждёт прогона)**
- `startShare`: на Windows (`cap.systemLoopback`) при «со звуком» видео публикуется БЕЗ getDisplayMedia-аудио (`audio:false`), а звук — отдельным треком: `createNativeAudioTrack({})` → `new LocalAudioTrack(track, undefined, true)` → `publishTrack(..., { source: ScreenShareAudio, name: 'screen-audio' })`. Не-Windows → прежний getDisplayMedia-путь.
- Teardown в `lib/livekit.ts` (`registerNativeScreenAudio`/`stopNativeScreenAudio`): снятие на `stopShare`, на остановке из ОС-бара (`LocalTrackUnpublished(ScreenShare)`), на выходе из комнаты (`disposeRoom`) — везде идемпотентно, останавливает и Rust-стрим. Заменяет ненадёжный `getDisplayMedia({audio:true})` (закрывает T-050a на Windows).
- Зрители получают звук бесплатно: `applyScreenSubscription` подписывает ScreenShareAudio при «смотреть», `applyVolumeFor` даёт громкость-на-слушателя. `typecheck` чистый.
- **Известное ограничение:** системный loopback ловит ВЕСЬ вывод, включая голоса других участников из колонок → возможен эхо-фидбек (как и раньше с getDisplayMedia; решается наушниками). Настоящее решение — выбор процесса (Stage B уже готов, нужен пикер в share-флоу) — отдельное улучшение.

## Подсказки

- Realtime-аудио НЕ проверяется юнит-тестами: ресемплинг 48k/44.1k, раскладка каналов, underrun/overrun, MMCSS-поток. Цикл — пишу → прогон на железе → правлю.
- `windows` crate уже добавлен (0.61, features `Win32_Foundation`/`Win32_System_Com`/`Win32_Media_Audio`) — Stage B (process loopback) переиспользует его, понадобится ещё `ActivateAudioInterfaceAsync` (async COM completion handler).
- WAVEFORMATEXTENSIBLE — `#[repr(packed)]`: поля (например SubFormat) читать только через `read_unaligned`, иначе E0793.
- **PROPVARIANT с VT_BLOB → STATUS_HEAP_CORRUPTION:** у `windows::…::PROPVARIANT` есть Drop → PropVariantClear → CoTaskMemFree(pBlobData). Если pBlobData указывает на стек (наши activation params), это free() стека = крэш. Оборачивать в `ManuallyDrop` (heap-памятью он не владеет). Поймали ручным прогоном Stage B, не компилятором.
- Порог 19041 легко поднять до 20348, если поймаем неработающие ранние Win10-билды (одна константа в `audio/mod.rs`).
