# T-084: Бэкап-скрипт + cron + восстановление

**Phase:** 5 — Production readiness
**Model:** Sonnet 4.6 · `medium`
**Estimate:** ~2-3 часа

## Контекст
Без бэкапов — никуда. Минимум для группы друзей: ежесуточный `pg_dump` + sync MinIO в отдельную локацию (другой VPS, S3 backup bucket, домашний NAS — что есть).

## Что сделать
1. **`backup.sh`** (запускается из docker-контейнера или через cron на VPS):
   ```bash
   #!/usr/bin/env bash
   set -euo pipefail
   
   TS=$(date +%Y%m%d-%H%M%S)
   BACKUP_DIR=/backups
   
   # PG
   docker exec kd-postgres pg_dump -U kakdela kakdela | gzip > $BACKUP_DIR/pg-$TS.sql.gz
   
   # MinIO via mc mirror
   docker run --rm -v $BACKUP_DIR:/backups -v $(pwd):/work \
     minio/mc:latest \
     /bin/sh -c "
       mc alias set src http://minio:9000 $S3_ACCESS_KEY $S3_SECRET_KEY &&
       mc mirror --overwrite src/kakdela /backups/minio-$TS/
     "
   
   # Rotation: keep last 7 daily, last 4 weekly, last 12 monthly
   find $BACKUP_DIR -name "pg-*.sql.gz" -mtime +7 -delete
   ```
   Адаптировать под docker network: если backup-контейнер в той же compose-сети, ссылается на сервисы по имени.
2. **`restore.sh`**:
   ```bash
   #!/usr/bin/env bash
   set -euo pipefail
   
   BACKUP=$1  # имя файла pg-YYYYMMDD-HHMMSS.sql.gz
   read -p "ВСЕ ТЕКУЩИЕ ДАННЫЕ БУДУТ ПЕРЕЗАПИСАНЫ. Введи 'YES' чтобы продолжить: " confirm
   [[ "$confirm" == "YES" ]] || exit 1
   
   gunzip -c /backups/$BACKUP | docker exec -i kd-postgres psql -U kakdela kakdela
   
   # MinIO restore тоже...
   ```
3. **Cron в compose** (новый сервис `backup-cron`):
   ```yaml
   backup:
     image: alpine:3
     volumes:
       - ./ops/backup:/scripts:ro
       - /var/run/docker.sock:/var/run/docker.sock:ro
       - kd-backups:/backups
     entrypoint: |
       /bin/sh -c "
         apk add --no-cache docker-cli &&
         echo '0 4 * * * /scripts/backup.sh' | crontab - &&
         crond -f
       "
   ```
4. **Off-site backup**: добавить шаг в backup.sh — rsync на удалённый сервер или upload в внешний S3. Пример:
   ```bash
   rsync -avz $BACKUP_DIR/ user@backup-host:/data/kakdela/
   # ИЛИ
   aws s3 sync $BACKUP_DIR s3://my-offsite-bucket/kakdela/
   ```
5. **`big-cheese backup`** — для ручного запуска перед опасными миграциями.

## Out of scope
- Point-in-time recovery (WAL archiving) — overkill для 15-20 людей.
- Encrypted backups — желательно если off-site, но не блокер.

## Подсказки
- pg_dump custom format (`-Fc`) удобнее для частичного restore, но `.sql.gz` проще и человекочитаем. Выбор за тобой.
- НЕ забудь добавить backup-volume в `docker-compose.prod.yml`. Дефолтный путь `/var/lib/docker/volumes/...` — если VPS маленький, лучше mount на отдельный диск.
