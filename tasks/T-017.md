# T-017: WS gateway — hello / ready, broadcast msg.new

**Phase:** 1 — Текстовый дом
**Model:** Opus 4.7 · `high`
**Estimate:** ~3 часа

## Контекст
Это ядро realtime-функциональности. Все события (новое сообщение, presence, typing, voice.join) текут через единый WebSocket-канал.

Читать: `ARCHITECTURE.md` §4 (WS event protocol), `.claude/CONVENTIONS.md` (раздел WebSocket).

## Что сделать
### Серверная часть

1. **WS endpoint** на `/ws` (через `@fastify/websocket`):
   - При connect клиент должен в первом сообщении прислать `{t:'hello', token}` (НЕ передаём токен в querystring, иначе он попадёт в access-логи).
   - Валидируем токен, если плохой — `close(4401, 'unauthorized')`.
   - При успехе сервер отвечает `{t:'ready', user, servers: [...]}` — стартовый снапшот.
2. **`Connection`** класс на сервере:
   - Содержит `userId`, `ws`, `subscribedChannels: Set<string>`, `subscribedServers: Set<string>`.
   - `send(event: ServerEvent)`, `close(code, reason)`.
   - При ready подписывается на все каналы серверов user'а.
3. **Реестр** `connections: Map<userId, Connection[]>` — потому что один user может иметь несколько окон (десктоп + ноут).
4. **Redis pub/sub** для broadcast: `BROADCAST(channelId, event)` → `redis.publish('kakdela:channel:'+id, JSON.stringify(event))`. Каждый процесс speedy подписан и при получении пересылает в свои локальные сокеты, отфильтрованные по `subscribedChannels`.
   - Для MVP можно и без Redis (один процесс), но архитектурно подготовь — оборачивай в интерфейс `Broker`, имплементация `InMemoryBroker` для теста и `RedisBroker` для прод.
5. **После `POST /api/channels/:channelId/messages` (из T-016)** — вызвать `broker.publish('channel:'+channelId, {t:'msg.new', channelId, message})`.
6. **Ping/pong**: сервер шлёт `{t:'ping'}` каждые 25 секунд, клиент отвечает `{t:'pong'}`. Если нет ответа 2× — закрыть сокет.
7. **Авторизация конкретных событий**: при `{t:'typing', channelId}` от клиента проверять, что user — член сервера канала. Иначе игнорировать (не закрывать соединение — может быть legitimate race при удалении канала).

### Клиентская часть

8. **`ws.ts`** — `WsClient` класс:
   - `connect()` — открывает WebSocket, шлёт hello, ждёт ready.
   - `send(event: ClientEvent)`, `on(handler: (e: ServerEvent) => void)`.
   - Авто-reconnect с экспоненциальной задержкой (1s → 2s → 4s → ... → max 30s).
   - Парсинг через `ServerEventSchema.parse()` из ginzu (см. T-005) — если не парсится, логируем и пропускаем (не валим соединение из-за одного битого фрейма).
9. **`realtime/store.ts`** — Zustand:
   - `status: 'disconnected' | 'connecting' | 'connected' | 'reconnecting'`
   - `latency: number | null` (последний ping RTT)
   - Подписка из других store'ов через `wsClient.on(...)`.
10. **App.tsx** — после auth подключаем WS, при unmount — закрываем.

## Out of scope
- Resume после disconnect с догрузкой пропущенных событий — фаза 5. Сейчас при reconnect клиент просто переподписывается и делает refetch активного канала через REST.
- Бинарные WS-фреймы / msgpack — не нужно, JSON хватит.
- Sticky session не нужен для одной инстансы speedy.

## Подсказки
- Не используй `nodejs:ws` напрямую — `@fastify/websocket` обёртка интегрирована с Fastify-логером и hooks.
- Redis pub/sub НЕ поддерживает acknowledgements — если процесс speedy упал между receive и forward, событие потеряется. Это OK для MVP, но для T-064 (DM с гарантией доставки) понадобится outbox-pattern.
- Клиентский reconnect с jitter: `delay = min(30s, 1s * 2^attempt) + random(0, 1000ms)`. Без jitter — все клиенты переподключаются одновременно после ребута сервера.
