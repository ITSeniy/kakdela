# T-087: Звонок и демо экрана в DM

**Phase:** 5 — Полировка
**Model:** Opus 4.7 · `high` (WebRTC, race conditions на 1-to-1 invite)
**Estimate:** ~4-5 часов

## Контекст

Сейчас в DM есть только текстовый чат — `DmScreen.tsx` не имеет action-кнопок в шапке. Дизайн (`designs/final-dm.jsx:170-188`) рисует три кнопки рядом с именем собеседника: «позвонить» (голос), «видео» (камера, требует T-082), «демо экрана».

LiveKit уже умеет всё, что нужно — звонок в DM это просто voice room с deterministic `room` name (`dm-${dmChannelId}`) и автоматическим invite-сигналом второй стороне через WS.

Читать: `tasks/T-030.md` (Guido), `tasks/T-031.md` (voice/join), `tasks/T-033.md` (useVoiceRoom), `tasks/T-064.md` (DM модель), `ARCHITECTURE.md` §3.2.

## Цель

В DM с любым собеседником можно начать voice-call, включить демо экрана, и второй стороне приходит invite-уведомление с возможностью принять или отклонить.

## Что сделать

1. **Backend room helper** в `guido.ts`:
   - `roomNameForDm(dmChannelId): 'dm-${id}'`. Стабильное имя — оба клиента попадают в одну комнату.
2. **`POST /api/voice/dm/:dmChannelId/join`**:
   - Проверка: user — один из двух participants DM. Иначе 403.
   - Выдача токена как в `T-031`, но с `room = dm-${dmChannelId}`.
   - Если в комнате ещё нет никого — broadcast `{t:'dm.call-invite', dmChannelId, fromUserId}` второй стороне.
   - Если вторая уже в комнате (она инициатор) — не шлём invite, просто join.
3. **WS events**:
   - `dm.call-invite` — отправляется только targeted userId (а не всем подписчикам канала). Расширить `broker` для personal-routing.
   - `dm.call-cancel` — инициатор отменил до того, как принято.
   - `dm.call-decline` — приёмник нажал «отклонить» (LiveKit участник удалён через `revokeUser`).
4. **DmScreen action buttons** (по дизайну `final-dm.jsx:170-188`):
   - Иконка «позвонить» (PhoneIcon) → `useVoiceRoom().joinDm(dmChannelId)`. Self-сторона сразу видит свой ParticipantTile, ждёт второго.
   - Иконка «видео» — заглушка с tooltip «нужна камера, см. T-082».
   - Иконка «демо» → join + сразу `startShare()`.
   - Все три **disabled** если уже в активном voice channel в другом месте.
5. **IncomingCall.tsx**:
   - Получает `dm.call-invite` через `wsClient.on`.
   - Toast в углу окна: «<displayName> звонит» + кнопки «принять» (→ join DM voice) и «отклонить» (→ POST cancel).
   - Звуковой сигнал (короткий `ring.mp3` в `public/`).
   - Auto-dismiss через 30 секунд (другая сторона получает timeout-cancel).
6. **useVoiceRoom — kind: 'dm'**:
   - Параметр `kind: 'channel' | 'dm'` для `joinChannel`.
   - При `'dm'` — токен от `/voice/dm/:id/join` вместо `/voice/:id/join`.
   - Layout в `VoiceScreen` тот же, но header показывает имя собеседника, не имя канала.

## Статус (2026-06-28, ветка feat/phase-6-mobile)

**Реализовано и закоммичено** в рамках причёсывания мобилки. Решения по ходу:
- Room name для DM — `dm-${channelId}` (отдельный неймспейс; webhook `media/webhook.ts`
  игнорит `dm-`, состав 1:1-комнаты ведёт LiveKit на клиенте — серверный
  presence-broadcast не нужен).
- «Кто инициатор» определяется по **LiveKit-truth** (`listDmParticipants`), а не
  Redis — нет залипшего presence от упавшего клиента. Pending-инвайт + 30s-таймер
  живут в памяти процесса (self-host = один инстанс speedy).
- read/typing звонка не трогаем; «видео» осталось заглушкой (нужна камера, T-082).
- Звонок ведёт `DmCallScreen` (лёгкая 1:1-версия `VoiceScreen`), входящий —
  `IncomingCall` (тост в углу, НЕ модалка). Док (`VoiceDock`) сделан DM-aware:
  держит звонок при переходах между чатами и телепортит обратно.

### Доработки (ветка feat/phase-6-polish, 2026-06-28)
- **Баг**: при выходе собеседника из активного звонка у второго экран откатывался
  в «Звоним…». Починено в `DmCallScreen` (`everConnectedRef`): уход после ответа =
  «звонок завершён», а не дозвон.
- **ПК always-on-top**: Rust-команда `set_call_alert` (поднять окно поверх всех +
  request_user_attention) дёргается из `IncomingCall` — тост видно, даже когда окно
  свёрнуто. *Возможное продолжение: отдельное маленькое окно-попап вместо подъёма
  главного.*
- **Call-log в чате**: на `room_finished` DM-комнаты webhook пишет системное
  сообщение «<имя> начал звонок, который продлился N» (колонка `messages.system`,
  миграция 0023; `SystemEvent` в ginzu; рендер `SystemLine` в `DmBubbleList`).
  Неотвеченный звонок не логируется.

### Доработки-2 (ветка feat/phase-6-polish, 2026-06-28)
- **Время под «хвостом» группы**: таймстамп вклинивался между склеенными
  сообщениями одного автора (висел под «головой»). `DmBubbleList` теперь считает
  флаг `tail` (последнее сообщение в склейке) и показывает время-метку только
  под ним; аватар по-прежнему у «головы».
- **Задержка call-log убрана**: `room_finished` LiveKit шлёт лишь спустя
  empty_timeout (минуты) — отсюда «сильная задержка». Теперь webhook фиксирует
  звонок на `participant_left`/`participant_connection_aborted`, как только
  комната опустела (проверка по LiveKit-truth `listDmParticipants`); `room_finished`
  остался подстраховкой. Дедуп двойного лога — через `del` ключа в `finalizeDmCall`.
- **ПК: отдельное окно-попап** вместо подъёма главного окна. Rust-команды
  `open_call_popup`/`close_call_popup` создают маленькое borderless always-on-top
  окно (label `call-popup`) в правом верхнем углу монитора. **Содержимое — НЕ
  SPA**, а самодостаточная статичная страница `public/call-popup.html` (без
  React/Vite-бандла/авторизации — отдельный webview не «провисает» на загрузке
  бандла, как было с белым окном на первой версии). Данные звонка едут в query
  `?cp=<base64url JSON>` (читаются синхронно из `window.location`). Кнопки шлют
  глобальный tauri-event `call-popup-action` (через `window.__TAURI__`,
  `withGlobalTauri:true`), который ловит `IncomingCall` (главное окно исполняет
  join/decline и закрывает попап); попап ещё и сам себя закрывает (после ответа
  и по таймауту-подстраховке 34с). Хост-обёртка `lib/host/call-window.ts`
  (open/close/onAction); capability расширена на окно `call-popup` +
  `core:window:allow-close`. Старый `set_call_alert` удалён. Гейт: typecheck +
  web build + speedy-тесты (44) + cargo check (desktop) зелёные.

## Acceptance criteria (DoD)

- [x] User-A кликает «позвонить» в DM → User-B видит toast «звонит» в углу окна
- [x] User-B «принять» → оба в voice-комнате, аудио идёт *(код; ручной A→B прогон ниже)*
- [x] User-B «отклонить» → у User-A toast «отклонил звонок»
- [x] User-A отменяет звонок до ответа → у User-B toast пропадает
- [x] Демо экрана в DM работает идентично server-каналу (кнопка «экран» = join + share)
- [x] Второй DM (с третьим user'ом) во время активного звонка — кнопки disabled
- [x] `pnpm typecheck` чистый (+ web build, speedy-тесты зелёные)
- [ ] **Ручная проверка:** на двух эмуляторах/машинах звонок проходит за < 2 сек
      (аудио туда-обратно, отклонение, отмена, таймаут 30с) — не выполнено,
      требует двух клиентов.

## Out of scope

- Групповые звонки в DM (3+ человек) — нет, для этого voice-каналы серверов.
- История звонков (call log) — нет.
- Voicemail — нет.
- Видео с камеры — нужна T-082 (camera track в LiveKit).

## Подсказки

- LiveKit Room name для DM **должен** включать prefix `dm-` — иначе можно случайно зайти в room `dm-<uuid>` мимо permission-check. В `/voice/:channelId/join` запретить serverless room names ручной валидацией.
- Не используй WS-broadcast по `subscribedChannels` для invite — это публичный канал, а звонок privately targeted. Расширь `broker.publishToUser(userId, event)`.
- Auto-cancel при timeout: серверный setTimeout на 30s от момента invite. При фактическом join — clearTimeout.
- IncomingCall — это **не модалка** (mode='dialog'). Не блокируй ввод в чате; toast в углу.
