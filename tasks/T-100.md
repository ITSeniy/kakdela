**Сделано — нативная часть (тулчейн установлен, APK собран и запущен 2026-06-27):**
- `tauri android init` → `gen/android/` сгенерирован и закоммичен (~42 файла скаффолда).
- `src-tauri/src/lib.rs`: десктопная обвязка (трей/меню/close-to-tray/global-shortcut) вынесена в `setup_desktop()` под `#[cfg(desktop)]`; `focus_main_window`/`set_tray_badge` на mobile — no-op. `cargo check` (desktop) зелёный — desktop не сломан.
- `AndroidManifest.xml`: +`RECORD_AUDIO`, +`CAMERA` (к `INTERNET`).
- **APK собран** (`--target x86_64`, debug) → **установлен и запущен на эмуляторе Pixel 10 Pro** → рендерит UF КакДела (экран входа) без крашей. WebView + Tauri-плагины работают.
- `docs/MOBILE.md`: проверенные команды сборки/install + грабли (ABI x86_64 для эмулятора, debug-`.so` ~150МБ → wipe-data, MSYS_NO_PATHCONV, JDK 17).

**Сделано — хранилище токена (2026-06-27):**
- `lib/host/secrets.ts` переписан: persistent encrypted-at-rest стор (non-extractable AES-GCM ключ в IndexedDB, шифрует значения; fallback на sessionStorage). Не localStorage. Переживает холодный старт.
- `auth/api.ts`: сессия (user+token) хранится в сторе под `kd:session`; `initAuth` оптимистично восстанавливает её (мгновенный старт + офлайн-устойчивость), затем валидирует через refresh-cookie. Стор теперь реально читается, а не write-only.
- Логин на эмуляторе подтверждён (адрес `10.0.2.2:3001`).

**Решение по hardware Keystore:** настоящий Android Keystore (EncryptedSharedPreferences/нативный мост) **сворачивается в T-101** — он load-bearing для sealed identity-key libsignal, строим один нативный мост на всё (крипто-ключ + опц. перенос токена). Софтварное шифрование at-rest закрывает «не в localStorage» уже сейчас.

**Осталось по T-100:**
- Прогон WS/звонка с **запущенным backend** на эмуляторе (DoD 5,6).
- getUserMedia в WebView (`onPermissionRequest`) — риск голоса, проверяется при T-087.

## Контекст

Tauri 2 умеет собирать под Android (`tauri android init`). `polly` уже изолирован от Tauri через `packages/polly/src/lib/host/` — это и есть наш билет на мобилку без переписывания бизнес-логики. Эта карточка — **только фундамент**: собрать APK, адаптировать shell под телефонный viewport, дописать Android-ветку в host-layer. **Никакого крипто здесь нет** — оно в T-101+.

Цель мобильного клиента — **личный мессенджер 1:1, не «мобильный Discord»**. На мобиле **нет** просмотра серверов, каналов, серверных голос-комнат и демо экрана. Единственная поверхность, общая с desktop-версией, — **незашифрованные cloud-DM** (T-064). Плюс секретные чаты (T-101…T-103), которые живут только на устройстве. Голос на мобиле — это **звонок 1:1 внутри DM** (LiveKit room `dm-${id}`, см. T-087), а не серверная voice-комната.

Читать: `ARCHITECTURE.md` §Tauri isolation, `.claude/CONVENTIONS.md` (раздел Desktop shell), `packages/polly/src/lib/host/*`.

## Что сделать

1. **Инициализация таргета**:
   - `tauri android init`, зафиксировать `gen/android` в репо (или в `.gitignore` по решению — описать в карточке выбор).
   - В `tauri.conf.json` — `identifier` уже `com.kakdela.polly`; добавить android-конфиг и dev-подпись (debug keystore).
   - Документировать требования к окружению (JDK 17, Android SDK, NDK, `cargo-ndk`) в `docs/MOBILE.md` (новый, короткий).
2. **Детект платформы в host-layer**:
   - `host/index.ts` экспортирует `platform: 'desktop' | 'android' | 'web'` и `isMobile`.
   - Бизнес-код ветвится **только** через host, не через `@tauri-apps/*` напрямую (CONVENTIONS — Tauri isolation).
3. **JWT в Android Keystore**:
   - `keychain.ts` — на android использовать Keystore-backed хранилище (через Tauri-плагин или Android-команду). На desktop остаётся stronghold, в web — `sessionStorage`. **Никакого `localStorage`** (CONVENTIONS).
4. **Мобильный shell**:
   - `MobileShell.tsx`: bottom-tab навигация (чаты · звонки · профиль), безопасные зоны (`env(safe-area-inset-*)`), single-column layout.
   - Существующие экраны DM (`features/dm/*` из T-064) рендерятся внутри мобильного layout — список во весь экран, чат во весь экран, назад по back-кнопке.
   - **Никакого server-rail и списка каналов** — bottom-nav это весь верхний уровень навигации. Сервера/каналы на мобиле не открываются.
   - На desktop ничего не меняется — `MobileShell` подключается только при `isMobile`.
5. **Уведомления**:
   - `notify.ts` android-ветка через локальные нотификации Tauri. Полноценный FCM-push — **stub + TODO** (отдельной задачей, перекликается с пропущенным T-067).
6. **Tray**: `tray.ts` — на мобиле no-op, без падений.
7. **Разрешения Android**: в `AndroidManifest` — `INTERNET`, `RECORD_AUDIO`, `CAMERA` (под будущий голос/видео в DM, T-087).

## Out of scope

- Крипто / секретные чаты — T-101, T-102, T-103.
- **Просмотр серверов / каналов на мобиле — нет.** Мобилка — личный мессенджер (cloud-DM + секретные чаты), не «мобильный Discord».
- **Серверные голос-комнаты на мобиле — нет** (только 1:1 звонки в DM, T-087).
- iOS — нет (Windows-first продукт, телефон — Android-first).
- Полный FCM-push — stub, отдельная задача.
- Планшетный / landscape layout — нет.
- Демо экрана с телефона — нет (это desktop-фича).

## Подсказки

- `tauri android init` тянет много нативного тулчейна — `docs/MOBILE.md` сэкономит часы следующему.
- getUserMedia в Android System WebView требует обработки `onPermissionRequest` на нативной стороне — это главный риск голоса на мобилке, проверить рано.
- Не плодить мобильные копии экранов: переиспользуй `features/dm/*`, оборачивая их в `MobileShell`. Расхождение desktop/mobile — только в layout-обёртке.
- Safe-area: айфоноподобной чёлки на Android нет, но навбар/жесты снизу есть — `env(safe-area-inset-bottom)` обязателен для composer'а.
